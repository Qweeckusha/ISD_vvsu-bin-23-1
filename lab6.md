# Лабораторная работа №6 – Нефункциональные требования
## ИС проектной организации (Вариант 5)

> **Немного теории для понимания**: 
> Если функциональные требования отвечают на вопрос *"Что система должна делать?"*, то нефункциональные – *"Как хорошо она должна это делать?"*. 
> Проще говоря: ФТ – это "система должна показывать список сотрудников", а НФТ – "этот список должен загружаться не дольше 2 секунд даже при 10 000 записей".

## Блок 1: Производительность (ПРОИЗ)

| ID | Нефункциональное требование | Уточнения |
|----|----------------------------|-----------|
| НФТ-ПРОИЗ-01 | Система должна отвечать на стандартные запросы (поиск сотрудника, просмотр договора) не более чем за **3 секунды** | Чтобы пользователь не скучал и не думал, что всё зависло |
| НФТ-ПРОИЗ-02 | Формирование сложных аналитических отчётов (например, ФТ-ОТЧ-12) не должно занимать более **15 секунд** | Тяжёлые запросы – это база, но 15 секунд – много |
| НФТ-ПРОИЗ-03 | Система должна поддерживать одновременную работу **до 50 пользователей** без деградации производительности | Примерно столько сотрудников могут работать в проектной организации одновременно |
| НФТ-ПРОИЗ-04 | Время загрузки главной панели управления не должно превышать **3 секунд** | Первое впечатление важно — пусть система "летает" при старте |

## Блок 2: Безопасность и защита данных (БЕЗ)

| ID | Нефункциональное требование | Уточнения |
|----|----------------------------|-----------|
| НФТ-БЕЗ-01 | Все пароли пользователей должны храниться в хешированном виде с использованием современных алгоритмов | Даже если базу украдут — злоумышленник не получит чужие пароли |
| НФТ-БЕЗ-02 | Сессия пользователя должна автоматически завершаться после **15 минут неактивности** | Забыл выйти из системы на общем компьютере? Система подстрахует |
| НФТ-БЕЗ-03 | Все соединения с системой должны осуществляться по защищённому протоколу **HTTPS** | Чтобы никто "по пути" не перехватил данные о договорах и зарплатах, и в целом ПДн |
| НФТ-БЕЗ-04 | Система должна блокировать учетную запись после **5 неудачных попыток входа** на 10 минут и отправлять отчёт безопаснику | Защита от подбора паролей "методом тыка" |
| НФТ-БЕЗ-05 | Конфиденциальные/персональные данные должны быть защищены в соответствии с **152-ФЗ** | Потому что за нарушения могут оштрафовать (заблокировать/замедлить/признать экстремисткой на территории РФ/отключить интернет глобально), и это не шутки |

## Блок 3: Интерфейс и удобство использования (ИНТ)

| ID | Нефункциональное требование | Уточнения |
|----|----------------------------|-----------|
| НФТ-ИНТ-01 | Интерфейс системы должен быть интуитивно понятным: новый пользователь должен освоить базовые операции за **не более 30 минут** | Без толстых инструкций – чтобы можно было просто найти методом "тыка" |
| НФТ-ИНТ-02 | Все формы ввода должны иметь валидацию данных с понятными сообщениями об ошибках | Не "Ошибка e4-xcV12jop237-1", а "Пожалуйста, введите корректный email" |
| НФТ-ИНТ-03 | Система должна корректно отображаться на экранах с разрешением от **1920×1080** и выше (4К, 500Гц, 0.01мс) | Чтобы работало и на ноутбуке, и на игровом мониторе в офисе начальника |
| НФТ-ИНТ-04 | Ключевые действия (сохранить, удалить, отправить) должны быть доступны не более чем в **3 клика** от главного экрана (конверсия) | Минимум кликов — максимум эффективности |
| НФТ-ИНТ-05 | Система должна поддерживать горячие клавиши для часто используемых операций (Ctrl+S, Ctrl+F и т.д.) | Для тех, кто любит работать быстро и без мышки |

## Блок 4: Надёжность и доступность (НАД)

| ID | Нефункциональное требование | Уточнения |
|----|----------------------------|-----------|
| НФТ-НАД-01 | Система должна быть доступна **не менее 99% времени** в рабочее время (09:00–18:00, Пн–Пт) | Примерно 4 часа "простоя" в месяц — допустимый компромисс |
| НФТ-НАД-02 | При сбое системы все несохранённые данные в формах должны восстанавливаться после перезагрузки страницы | Чтобы не пришлось заново вводить полдоговора из-за моргнувшего света |
| НФТ-НАД-03 | Система должна корректно обрабатывать ситуации потери соединения с сервером, информируя пользователя | Не просто "всё сломалось", а "проверьте подключение и попробуйте снова" |
| НФТ-НАД-04 | Критические ошибки не должны приводить к полному падению системы — остальные модули должны продолжать работать | Если "отчётность" упала — "учёт сотрудников" всё равно должен работать |

## Блок 5: Сопровождаемость и развитие (СОПР)

| ID | Нефункциональное требование | Уточнения |
|----|----------------------------|-----------|
| НФТ-СОПР-01 | Код системы должен быть документирован: каждая функция/модуль имеет описание назначения и параметров | Чтобы следующий разработчик не гадал, "эт чё?" |
| НФТ-СОПР-02 | Система должна иметь модульную архитектуру, позволяющую добавлять новые типы сотрудников или отчёты без переписывания ядра | Сегодня — конструкторы, завтра — дизайнеры: система должна быть готова к изменениям |
| НФТ-СОПР-03 | Все изменения в структуре данных должны сопровождаться миграционными скриптами | Чтобы при обновлении системы старые данные не потерялись и не "поехали" |
| НФТ-СОПР-04 | Система должна предоставлять API для интеграции с внешними системами (типа LLMки или других инструментов) | Вдруг понадобится "подружить" нашу систему с чем-то ещё |

## Блок 6: Портативность и развёртывание (ПОРТ)

| ID | Нефункциональное требование | Уточнения |
|----|----------------------------|-----------|
| НФТ-ПОРТ-01 | Система должна развёртываться на ОС семейства **Linux** и **Windows Server** | Чтобы не зависеть от одной платформы и выбрать то, что уже есть в организации |
| НФТ-ПОРТ-02 | Развёртывание системы должно быть возможно с помощью скрипта автоматической установки (Docker / Ansible) | Чтобы не настраивать всё вручную по 3 часа, а запустил скрипт — и всё готово |
| НФТ-ПОРТ-03 | Система должна корректно работать с СУБД **PostgreSQL 13+** и **MySQL 8+** | Гибкость в выборе базы данных — вдруг в организации уже есть лицензия на одну из них |
| НФТ-ПОРТ-04 | Клиентская часть должна работать в современных браузерах (Chrome, Firefox, Edge последних версий) | Без экзотики (Zen Browser) — только то, что реально стоит у сотрудников |

---

## Блок 7: Целостность и резервирование данных (ДАН)

| ID | Нефункциональное требование | Уточнения |
|----|----------------------------|-----------|
| НФТ-ДАН-01 | Система должна выполнять автоматическое резервное копирование базы данных **ежедневно в 02:00** | Чтобы в случае "о неет, мой дом, всё пропало" можно было восстановиться со вчерашней версии |
| НФТ-ДАН-02 | Резервные копии должны храниться **не менее 30 дней** с возможностью выборочного восстановления | Не просто "бэкап есть", а "можно восстановить только один удалённый договор" |
| НФТ-ДАН-03 | Все операции изменения критических данных (удаление сотрудника, изменение стоимости договора) должны выполняться в транзакциях | Чтобы не получилось: деньги списались, а договор не создался — атомарность, как в банке |
| НФТ-ДАН-04 | Система должна проверять целостность ссылок между объектами (например, нельзя удалить отдел, в котором есть сотрудники) | Защита от "случайного нажатия" и логических ошибок в данных |

## Чек-лист: всё ли учли?

- [x] Производительность: отклик системы, время формирования отчётов, нагрузка
- [x] Безопасность: пароли, сессии, шифрование, соответствие 152-ФЗ
- [x] Удобство: интуитивный интерфейс, валидация, горячие клавиши
- [x] Надёжность: доступность 99%, восстановление после сбоев, изоляция ошибок
- [x] Сопровождаемость: документация, модульность, миграции, API
- [x] Портативность: кроссплатформенность, контейнеризация, поддержка СУБД
- [x] Данные: бэкапы, транзакции, целостность связей
